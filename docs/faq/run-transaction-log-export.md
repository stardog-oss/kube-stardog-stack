# Run a Transaction Log Export

Transaction logs can be used with database backups for point-in-time recovery. This article shows how to inspect and export a Stardog database transaction log from a Kubernetes deployment.

Do not save exported transaction logs on the Stardog data PVC. Store exported transaction log files on the backup volume or remote backup storage so they survive data-volume replacement during a restore.

For server backup storage setup, see [Run a server backup](run-server-backup.md).

## Prerequisites

- A running Stardog deployment.
- A database with transaction logging enabled.
- A backup volume mounted into the Stardog pod, or another backup location outside the Stardog data PVC.
- A user with permission to run Stardog admin operations for the target database.

In Stardog Cluster deployments, transaction logging is enabled automatically and is required for replication. For standalone deployments, enable transaction logging when creating the database or while the database is offline.

## Assumptions

This runbook uses example values for the environment-specific names:

- Namespace: `stardog`
- Helm release: `sd-stack`
- Database: `db360`
- Stardog pod: `stardog-sd-stack-0`
- Backup PVC: `stardog-backup-output`

Replace those values with the names from your environment.

The remaining names and paths follow the default `kube-stardog-stack` behavior for that release:

- Stardog StatefulSet: `stardog-<release>`
- Stardog pod: `stardog-<release>-0`
- Backup volume mount inside the Stardog pod: `/var/opt/stardog/backups`
- Backup path generated by the chart: `/var/opt/stardog/backups/<backupDir>/<release>/<node>`

For the examples in this runbook, `<release>` is `sd-stack`.

## Confirm the Backup Volume

When `backup.enabled=true` and the backup location is an Azure or persistent-volume location, the chart mounts the backup volume at `/var/opt/stardog/backups`.

Set the working variables:

```bash
export NAMESPACE=stardog
export RELEASE=sd-stack
export POD=stardog-sd-stack-0
export DB=db360
export TXLOG_BACKUP_ROOT=/var/opt/stardog/backups/transaction-logs
```

Verify the pod and backup mount:

```bash
kubectl -n "$NAMESPACE" get pod "$POD"
kubectl -n "$NAMESPACE" exec "$POD" -- df -h /var/opt/stardog/backups
```

If `/var/opt/stardog/backups` is not mounted from backup storage, stop here and configure backup storage first. Do not use `/var/opt/stardog/txlog-*`, `/var/opt/stardog/data`, or another path on the Stardog data PVC as the long-term export target.

## Enable Transaction Logging

For Stardog Cluster, transaction logging is already enabled and cannot be disabled. Therefore you can skip this step.

For a new standalone database:

```bash
kubectl -n "$NAMESPACE" exec "$POD" -- \
  stardog-admin db create -o transaction.logging=true -n "$DB"
```

For an existing standalone database, take the database offline before changing the setting:

```bash
kubectl -n "$NAMESPACE" exec "$POD" -- stardog-admin db offline "$DB"
kubectl -n "$NAMESPACE" exec "$POD" -- \
  stardog-admin metadata set -o transaction.logging=true -- "$DB"
kubectl -n "$NAMESPACE" exec "$POD" -- stardog-admin db online "$DB"
```


## Inspect the Transaction Log

List databases and confirm the target database exists:

```bash
kubectl -n "$NAMESPACE" exec "$POD" -- stardog-admin db list
```

View the transaction log summary:

```bash
kubectl -n "$NAMESPACE" exec "$POD" -- stardog-admin tx log "$DB"
```

View individual updates:

```bash
kubectl -n "$NAMESPACE" exec "$POD" -- stardog-admin tx log "$DB" --updates
```

## Export the Transaction Log to Backup Storage

Create a transaction-log export directory on the backup volume:

```bash
export TXLOG_TIMESTAMP=$(date -u +%Y%m%dT%H%M%SZ)
export TXLOG_DIR="${TXLOG_BACKUP_ROOT}/${RELEASE}/${POD}/${DB}"
export TXLOG_FILE="${TXLOG_DIR}/${DB}-${TXLOG_TIMESTAMP}.txlog"

kubectl -n "$NAMESPACE" exec "$POD" -- mkdir -p "$TXLOG_DIR"
```

Export the raw transaction log:

```bash
kubectl -n "$NAMESPACE" exec "$POD" -- \
  stardog-admin tx log "$DB" --format raw --output "$TXLOG_FILE"
```

Confirm the file exists on the backup volume:

```bash
kubectl -n "$NAMESPACE" exec "$POD" -- ls -lh "$TXLOG_FILE"
kubectl -n "$NAMESPACE" exec "$POD" -- stardog-admin tx log --file "$TXLOG_FILE"
```

The exported file should remain on backup storage. Do not copy it into the Stardog data directory as the only retained copy.

## Optional Test Transaction

If you need to create a small transaction for validation, run a simple update against a non-production database:

```bash
kubectl -n "$NAMESPACE" exec "$POD" -- stardog query execute "$DB" \
  'INSERT DATA {
     GRAPH <urn:stardog:txlog:test> {
       <urn:stardog:txlog:test:doc-001> <urn:stardog:txlog:test:name> "Transaction log export test" .
     }
   }'
```

Then inspect or export the transaction log again.

## Point-in-Time Recovery Notes

Transaction log replay should be tested before it is needed in production:

```bash
kubectl -n "$NAMESPACE" exec "$POD" -- \
  stardog-admin tx replay --dry-run "$DB" "$TXLOG_FILE"
```

Only replay transaction logs against the intended restored database state. The log must contain the database's last committed transaction UUID from the backup point and complete transaction records after that point.

Avoid concurrent writes during point-in-time recovery. Time-based replay filtering requires `--skip-validate` and should be used carefully because it can bypass continuity validation.

## Cleanup

If you created the optional test graph, remove it after validation:

```bash
kubectl -n "$NAMESPACE" exec "$POD" -- stardog query execute "$DB" \
  'DELETE WHERE {
     GRAPH <urn:stardog:txlog:test> {
       ?s ?p ?o
     }
   }'
```

Keep the exported transaction log on backup storage according to the same retention policy as the database backup it complements.
